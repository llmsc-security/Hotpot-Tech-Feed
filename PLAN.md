# Hotpot Tech Feed — Plan v1

_Last updated: 2026-04-30_

A multi-tenant website + daily email service that tracks CS news, blogs, research papers, and star-level lab announcements; lets users subscribe to filtered feeds; and accepts open user contributions of new sources with quality controls.

---

## 2. Tech stack

- **Backend**: Python 3.11 + FastAPI
- **Worker queue**: Celery + Redis
- **Database**: Postgres 16 — items, users, subscriptions, contributions, votes, source health
- **Vector store**: Qdrant — embeddings for dedup + similarity
- **Embeddings**: `bge-m3` (multilingual), local on L40s
- **LLM**: Qwen3.6 via OpenAI SDK pointed at `api.ai2wj.com/v1`
- **Front-end**: React 18 + Vite + TanStack Query + Tailwind
- **Email**: Gmail SMTP (`jornbowrl@gmail.com` + app password)
- **Reverse proxy**: Caddy (auto-TLS) at `feed.ai2wj.com`
- **Raw cache**: filesystem object store for raw HTML so reparsing doesn't re-crawl

## 3. Architecture

Five logical services, one repo, separated by Celery task families:

1. **Ingestor** — one adapter per source family (RSS/Atom, arXiv API, HTML scraper, GitHub releases, Twitter/Nitter, WeChat aggregator, Xiaohongshu). Scheduled via Celery beat. Emits raw items to the normalizer queue.
2. **Normalizer + Dedup** — canonicalizes URLs, maps to common schema, runs three-stage dedup (URL → title fuzzy 7-day window → embedding cosine > 0.92). Writes to Postgres and Qdrant.
3. **Enricher** — Qwen calls for tagging (topic + content type), 1–2 sentence summary, AI commentary (prompt template keyed on content type).
4. **Feed engine** — for each user, materializes the daily digest based on subscriptions, ranks by `relevance × recency × source_trust`, caps at user's chosen size, injects 5–10% adjacent-topic items.
5. **Web + email** — React SPA hits FastAPI for browse/subscribe/contribute/vote; cron triggers email digest send.

```
[ Sources ]
     │
     ▼
[ Ingestor ]──► raw_queue ──► [ Normalizer + Dedup ]──► Postgres + Qdrant
                                                              │
                                                              ▼
                                                         [ Enricher ]
                                                              │
                                                              ▼
                                                       (tags, summary, commentary)
                                                              │
                                                              ▼

                                                          │       │
                                                          ▼       ▼

```

## 6. Tagging + grouping

Two orthogonal axes.

- **Topic** (curated top level; subfield tags free-form within): ML, Systems, Theory, Security, HCI, PL, DB, Networks, Graphics, Robotics.
- **Content type**: Paper, Blog post, News, Lab announcement, Tutorial, OSS release.

Process: Qwen tags every new item with topic(s) + content type + free-form subfield tags. Confidence stored. User overrides are recorded and weighted higher. Re-tag on user override only.

## 7. Dedup

Three stages, cheapest first:

1. **Canonical URL**: strip UTM params, normalize, match.
2. **Title fuzzy match** within ±7 days from earliest seen (Levenshtein ratio ≥ 0.9 on lowercased + de-stopworded titles).
3. **Embedding cosine** ≥ 0.92 (using `bge-m3` over title + first 500 chars of excerpt).

Duplicates are **collapsed, not dropped** — one canonical item with a `also_published_at: [...]` list. The feed shows the canonical item; the UI lets users expand to see all sources.

## 8. Quality controls for contributions

- **Per-contributor reputation** — starts at 0, +1 per accepted contribution, −2 per rejected, ±votes from others.
- **Per-source health score** — auto-degraded if failure streak exceeds threshold or if items have low average rating.
- **Probation bucket** — new sources visible only to opt-in users for 7 days before promotion.
- **Flag → soft-hide** — N flags within 24 h auto-hides an item pending moderation review.
- **Spam classifier** — small Qwen prompt at submit time blocks obvious junk before it enters probation.
- **URL blocklist** for known-bad domains.

## 9. Subscription model

Per user, three subscription dimensions plus a persona preset:

- **Topic tags** (multi-select, weighted)
- **Content types** (multi-select)
- **Explicit star-labs / sources** (always include)
- **Persona preset**: Researcher / Engineer / Learner / Generalist — these are bundles of sensible defaults users can start from and tune.

Plus a global "surprise me" knob that controls 5–15 % out-of-bubble injection.

## 10. Email digest

- Sent daily at the user's local-morning time (per-user TZ).
- Format: title + link + 1–2 sentence Qwen summary + topic chips.
- If full content unfetchable: show title + link + "*Full text not available — visit source.*"
- Footer: one-click thumbs-up, thumbs-down, "show less from this source" — these post back via signed tokens (no login needed from the email).
- Sent via Gmail SMTP using app password. DKIM/SPF must be configured on `ai2wj.com` to avoid spam folder.

## 11. AI commentary

Each item gets one Qwen-generated commentary block at enrichment time, cached, regenerated only if the item changes. Four prompt templates keyed on content type:

- **Paper** — "What's the contribution? Who is this for? What recent work does it relate to?"
- **Blog post** — "What problem does this discuss? What's the core argument?"
- **News** — "What happened? Why does it matter?"
- **Lab announcement** — "What was announced? Who from this lab is involved?"

Each item exposes a "Ask Qwen a follow-up" button — opens a small chat scoped to that item, no thread/comment system to moderate.

> **Risk**: AI commentary on every item turns into slop fast if prompts are generic. Tune prompts on real items in Phase 1 before turning commentary on globally.

## 12. Phasing

**Phase 1 — single-user MVP (~2 weeks)**

- arXiv adapter, RSS adapter, ~10 seed sources
- Three-stage dedup
- Qwen tagger + summarizer (commentary off until tuned)
- React UI: browse + filter + per-item view
- Daily email digest via Gmail SMTP

**Phase 2 — multi-tenant (~2 weeks)**

- Email + password auth, email verification
- Per-user subscriptions and persona presets
- Star-lab curated source list seeded
- Per-user TZ + digest scheduling
- Vote / flag plumbing

**Phase 3 — contributions (~2 weeks)**

- Submit flow + moderation queue
- Reputation + probation bucket
- Source health scoring + auto-degrade
- Spam classifier at submit time

**Phase 4 — Chinese sources (~2 weeks)**

- WeChat aggregator integration
- Xiaohongshu adapter (best-effort)
- Bilingual UI (zh-CN locale)
- User-submitted WeChat URL fallback path

**Phase 5 — polish + scale**

- AI commentary turned on, prompt-tuned
- Better dedup (cross-language)
- "Surprise me" tuning + feed-quality metrics
- Source health dashboard
- Abuse handling, rate-limits

## 13. Known hard parts

- **WeChat / Xiaohongshu crawling** — anti-scraping is aggressive; expect partial coverage, plan a user-submission fallback. Budget Phase 4 generously.
- **Star-lab list maintenance** — curated lists go stale; quarterly review, plus a "suggest a lab" form into moderation.
- **AI commentary slop risk** — generic prompts produce generic text; mitigate via per-content-type prompt templates and a quality-rating loop.
- **Email deliverability** — Gmail SMTP works but reputation must be earned; configure SPF/DKIM/DMARC on `ai2wj.com`, send slowly at first.
- **Robots.txt and ToS** — respect by default; maintain explicit override list only for sources where scraping is confirmed OK.

## 14. Open items to verify in Phase 1

- Whether `api.ai2wj.com/v1` exposes an `/embeddings` endpoint. If yes, use it; if no, run `bge-m3` locally on an L40.
- Qwen3.6 latency for batch tagging (target: < 1s per item end-to-end).
- Pgvector vs Qdrant trade-off at expected scale (we picked Qdrant; revisit if it's overkill).
- Email deliverability via Gmail SMTP at digest scale (start ≤ 100/day; raise after warm-up).
